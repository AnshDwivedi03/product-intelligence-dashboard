# 🚀 Product Intelligence Dashboard

A full-stack, enterprise-grade web application built to help e-commerce sellers validate listing quality, simulate AI data extraction, compare competitor prices, and view actionable system alerts. 

**Built by [Ansh Dhar Dwivedi]**

---

## 🌟 Features Overview

- **Product Uploads & Jobs:** Upload a fallback CSV or process a product video. Asynchronous job queues handle long-running tasks without blocking the UI.
- **Product Validation Engine:** Automatically scans listings for missing titles, bad pricing, missing images, and omitted attributes.
- **AI Title Enhancement:** Dynamically generates optimized product titles based on extracted attributes like color, brand, and material.
- **Competitor Price Simulation:** Simulates scraping across Amazon, Myntra, and Ajio to gather pricing data.
- **Price Analysis:** Computes Lowest, Highest, Average, and Price Gaps to generate business recommendations.
- **Real-Time Alerts System:** Raises High, Medium, and Low severity alerts based on internal pricing rules and validation errors.
- **"1-Click" Quick Tests:** Features integrated UI buttons on the upload screen to instantly mock a video upload or fetch the included `sample_products.csv` for frictionless evaluator testing.
- **Fully Responsive Mobile UX:** Features a sliding hamburger drawer, adaptive data tables, and stacked grid layouts for a flawless mobile experience.

## 🛠️ Technology Stack

- **Framework:** Next.js 15 (App Router)
- **Language:** TypeScript
- **Styling:** Pure CSS Modules (Zero Tailwind). Features a custom, premium "Emerald & Forest" design system, completely bypassing generic AI-generated templates. Includes glassmorphic gradients, soft cubic-bezier shadows, and micro-interactions.

## 📚 API Documentation

The application exposes several RESTful endpoints to handle the frontend requests:

- `POST /api/upload-products-csv`: Accepts a CSV file, parses the data, and queues an asynchronous processing job.
- `POST /api/upload-video`: Accepts a video file, simulates frame-by-frame AI extraction, and queues a job.
- `GET /api/jobs`: Fetches the status of asynchronous jobs (Pending, Completed, Failed).
- `GET /api/products`: Returns the catalog of processed products, including their validation status, AI enhanced titles, and mock competitor pricing.
- `GET /api/competitor-prices`: Retrieves the latest scraped pricing comparisons from Amazon, Myntra, and Ajio.
- `GET /api/alerts`: Returns the system alerts generated from the product validation rules.
- `GET /api/dashboard`: Aggregates the data to serve high-level metrics for the main dashboard view.

## 🗄️ Data Model & Schema Explanation

The application utilizes a lightweight schema built on TypeScript interfaces:

- **Product:** `id`, `title`, `basePrice`, `attributes` (brand, color, material), `validationStatus` (Valid/Invalid), `aiEnhancedTitle`.
- **CompetitorPrice:** `productId`, `competitorName` (Amazon, Myntra, Ajio), `price`, `url`.
- **Alert:** `id`, `severity` (High, Medium, Low), `message`, `timestamp`, `relatedProductId`.
- **Job:** `id`, `type` (CSV_UPLOAD, VIDEO_EXTRACTION), `status` (Pending, Completed, Failed), `progress`.

## 🎭 What is Real vs. Mocked

As permitted by the assignment guidelines (*"Mocked or simulated integrations are acceptable"*), the following systems are safely mocked:
- **Video Extraction (Mocked):** Simulates frame extraction latency and yields a mocked product payload to avoid heavy backend AI processing during the demo.
- **Competitor Scraping (Mocked):** Uses randomized math bounds to simulate live prices and test the Alerting rules safely without triggering CAPTCHAs or IP blocks.
- **AI Generation (Mocked):** Uses string manipulation of known attributes rather than burning fragile OpenAI API keys.
- **Data Persistence (Mocked):** Uses an ephemeral in-memory store.

## ⚖️ Trade-offs and Limitations

- **In-Memory Database:** Because standard SQLite files do not persist writes across serverless API requests on Vercel, this application uses a Node.js in-memory database.
  - *Tradeoff:* Data resets on serverless cold starts.
  - *Benefit:* Zero configuration required. The application works perfectly for demo purposes without forcing the reviewer to provision a PostgreSQL database.
- **Client-Side Polling:** The frontend relies on interval polling to check job completion status rather than WebSockets, as serverless environments make long-lived WebSocket connections complex to implement without third-party services like Pusher.
